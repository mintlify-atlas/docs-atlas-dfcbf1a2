RedQuanta MCP enforces a strict command allowlist: only the tools explicitly registered in `config/allowedCommands.json` (or its Windows equivalent) can be executed. Every argument is validated before the command runs — arguments not in the allowlist are rejected outright, and dangerous arguments (those that enable OS detection, spoofing, brute-force persistence, or proxy routing) require `DANGEROUS_MODE=true`. This two-tier system lets you run safe reconnaissance operations without enabling the broader destructive flag.

## How argument validation works

When a tool call reaches the server, `ArgGuard` performs the following checks in order:

1. **Command allowlist** — the command name must exist in the `allowed` map. Any command not listed is immediately rejected with `Command not allowed: <name>`.
2. **`requiresDangerous` flag** — if the command entry sets `requiresDangerous: true`, the call fails unless `DANGEROUS_MODE=true`.
3. **Per-argument injection scan** — every argument is matched against denial patterns for command injection (`$(...)`, backticks, `&&`, `;`, pipe redirects), path traversal (`../`, URL-encoded variants), and shell metacharacters (`$VAR`, `%VAR%`, `*`, `?`, `~`).
4. **Argument allowlist** — flag arguments (those beginning with `-`) must appear in the command's `allowedArgs` list. Non-flag values such as target hosts are passed through.
5. **Dangerous argument gating** — if a flag appears in the command's `dangerousArgs` list and `DANGEROUS_MODE` is off, the call is rejected with `Dangerous argument <flag> requires --dangerous flag`.

On any rejection, the operation is blocked and an audit log entry is written. No partial execution occurs.

<Note>
  Non-flag values (target IPs, hostnames, file paths) are not checked against the argument allowlist, but they are still scanned for injection and traversal patterns.
</Note>

## Denied command-line patterns

In addition to denied command names, the following regex patterns are matched against the full constructed command string. A match in any argument causes the entire call to be rejected.

```json config/allowedCommands.json (denied.patterns)
[
  "rm\\s+-rf",
  "dd\\s+if=",
  "mkfs\\.",
  "format\\s+[A-Z]:",
  "del\\s+/[qsf]",
  "deltree",
  "shutdown",
  "reboot",
  "halt",
  "poweroff",
  "init\\s+[06]"
]
```

The separate `config/deniedPatterns.json` file defines additional pattern categories scanned against every argument:

| Category | Examples blocked |
|----------|-----------------|
| Command injection | `$(...)`, `` `...` ``, `\|\|`, `&&`, `;cmd`, `\|cmd`, `>file`, `eval(`, `exec(` |
| Path traversal | `../`, `..\`, `%2e%2e%2f`, `%252e%252e%252f`, URL-encoded variants |
| Shell metacharacters | `${VAR}`, `$VAR`, `%VAR%`, `*`, `?`, `[...]`, `~`, `!`, `#` |
| Sensitive data | Strings matching `password`, `token`, `api_key`, `jwt`, `bearer`, `authorization` |

## Customizing the allowlist

To add a new tool or extend the arguments for an existing one, edit `config/allowedCommands.json` (or `config/allowedCommands-windows.json` for Windows). The schema for each entry is:

```json config/allowedCommands.json
{
  "allowed": {
    "mytool": {
      "path": "mytool",
      "minVersion": "1.0",
      "allowedArgs": ["-a", "-b", "--verbose", "--output"],
      "dangerousArgs": ["--write", "--delete"],
      "requiresDangerous": false
    }
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `path` | string | Executable name or absolute path. |
| `minVersion` | string | Minimum acceptable version string (informational; not enforced at runtime). |
| `allowedArgs` | string[] | Flags permitted without `DANGEROUS_MODE`. Prefix wildcards are supported (e.g., `-p*`). |
| `dangerousArgs` | string[] | Flags gated behind `DANGEROUS_MODE=true`. |
| `requiresDangerous` | boolean | If `true`, the entire tool is blocked unless `DANGEROUS_MODE=true`. |

<Note>
  Changes to the configuration files take effect on the next server restart. No hot-reload is performed.
</Note>
