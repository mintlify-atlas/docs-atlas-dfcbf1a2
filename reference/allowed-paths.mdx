Every file operation in RedQuanta MCP passes through `PathGuard`, a validation layer that resolves paths to their canonical form and verifies they remain inside the configured jail directory. The jail is anchored to the `JAIL_ROOT` environment variable (`/tmp/redquanta` by default) and cannot be escaped by any combination of relative segments, URL encoding, or symlinks. Operations that would read or write outside this boundary are rejected before they reach the filesystem.

</Note>

## How path validation works

`PathGuard.validatePath(inputPath, allowWrite?)` applies the following checks in sequence:

1. **Environment variable expansion** â€” on Windows, `%VAR%` references are expanded before processing.
2. **Path normalization** â€” separators are normalized for the current platform.
3. **Canonicalization** â€” the path is resolved to an absolute canonical path using `path.resolve()`. Relative inputs are anchored to `JAIL_ROOT`.
4. **Jail boundary check** â€” `path.relative(JAIL_ROOT, canonicalPath)` must not begin with `..` and must not be an absolute path itself. Any path that would escape the jail returns `isValid: false` with reason `Path escapes jail root`.
5. **Traversal pattern scan** â€” the original input is checked against traversal patterns regardless of canonicalization result (catches encoded attacks before they are decoded):

   | Pattern | Example |
   |---------|---------|
   | `../` | `../../etc/passwd` |
   | `..\` | `..\windows\system32` |
   | `%2e%2e%2f` | URL-encoded `../` |
   | `%252e%252e%252f` | Double URL-encoded `../` |
   | `%5c` | URL-encoded `\` |
   | `%2f` | URL-encoded `/` |

6. **Denied extension check** â€” the file extension must not appear in `deniedExtensions`.
7. **Allowed extension check** â€” if `allowedExtensions` is non-empty, the extension must be in the list.
8. **Directory allowlist** â€” the first path component relative to `JAIL_ROOT` must be in `allowedDirectories`.
9. **File size check** â€” if the file already exists, its size must not exceed `maxFileSize` (100 MB).
10. **Write permission** â€” write operations require `allowWrite=true` and the configuration must not have `readOnlyByDefault: true`, or write operations must not be in the `requiresDangerousFlag` list.

If all checks pass, `validatePath` returns:

```typescript
{
  isValid: true,
  canonicalPath: "/opt/redquanta/vol/reports/scan.xml",
  relativePath: "reports/scan.xml",
  isWritable: false,   // true only if allowWrite=true and DANGEROUS_MODE is on
  size: 4096,          // bytes, if file exists
  exists: true
}
```

On any failed check, `isValid` is `false` and `reason` describes why. The operation is blocked and the rejection is written to the audit log.

<Warning>
  Write, delete, execute, modify, and create operations are gated behind `DANGEROUS_MODE=true`. In standard mode, all filesystem operations are read-only.
</Warning>

## What happens on an invalid path

When `validatePath` rejects a path:

1. The operation is immediately blocked â€” no read or write occurs.
2. An audit log entry is written to `logs/audit-YYYY-MM-DD.jsonl` with:
   - The original input path
   - The resolved canonical path (if available)
   - The rejection reason
   - The tool name and execution ID
3. The calling tool receives an error response describing why the path was rejected (e.g., `Path escapes jail root`, `File extension '.exe' is denied`, `Directory 'etc' is not in allowed list`).

The audit log format is newline-delimited JSON (JSONL). Each entry includes a timestamp, action type, and full context for forensic review.

## Customizing allowed paths for your deployment

To adjust the filesystem jail for your environment, edit `config/allowedPaths.json` (Linux/macOS) or `config/allowedPaths-windows.json` (Windows).

### Change the jail root

```json config/allowedPaths.json
{
  "jailRoot": "/data/pentest/jail",
  "jailRootAlternatives": [
    "/data/pentest/jail",
    "/tmp/redquanta"
  ]
}
```

You can also override the jail root at runtime without modifying the config file:

```bash
JAIL_ROOT=/data/pentest/jail node dist/server.js
```

### Add a custom subdirectory

Add the directory name to `allowedDirectories`. `PathGuard` creates it automatically on startup:

```json config/allowedPaths.json
{
  "allowedDirectories": [
    "tmp", "reports", "uploads", "downloads", "wordlists",
    "custom-scans"
  ]
}
```

### Allow additional file extensions

Add the extension (with leading dot, lowercase) to `allowedExtensions`. Make sure the same extension does not appear in `deniedExtensions`:

```json config/allowedPaths.json
{
  "allowedExtensions": [
    ".txt", ".json", ".xml", ".csv", ".log", ".md",
    ".nessus"
  ]
}
```

### Increase file size limits

```json config/allowedPaths.json
{
  "maxFileSize": "500MB",
  "maxTotalSize": "5GB"
}
```

<Note>
  Configuration changes take effect on the next server restart. The jail root directory and all allowed subdirectories are re-created on startup if they do not already exist.
</Note>
